AUDYT jadzia-core — Raport Kompleksowy
Data: 2026-06-02 | Scope: pełne repozytorium | Metoda: analiza wszystkich 60+ plików źródłowych
🔴 Poziom 1 — Krytyczne (natychmiastowe działanie)
S1. LIVE SECRETS w .env skomitowane do repo
Plik: .env:1-38  
Dowód: ANTHROPIC_API_KEY, SSH_HOST, SSH_USER, TELEGRAM_WEBHOOK_SECRET — wszystkie jawne w historii gita.  
Ryzyko: Każdy z dostępem do repo ma klucz API Anthropic (płatny), dane logowania SSH, secret webhooka Telegram.  
Zalecenie: Rotacja wszystkich sekretów → git rm --cached .env → dodać .env do .gitignore → git filter-branch lub BFG Repo-Cleaner do usunięcia z historii.
S2. Divergentne wersje dependencji
- pyproject.toml:7 → anthropic>=0.18.0
- requirements.txt:5 → anthropic>=0.40.0  
To samo dla fastapi, uvicorn, httpx, pydantic, cachetools.
Ryzyko: CI używa requirements.txt, ale pyproject.toml jest source of truth dla deweloperów. Prowadzi do "works on my machine".
S3. jadzia.service → User=root
Plik: deployment/jadzia.service:8-9  
Ryzyko: Cała aplikacja działa jako root. W przypadku exploita (np. przez webhook/czytanie plików SSH) atakujący ma pełną kontrolę nad VPS.  
Zalecenie: Dedicated system user jadzia z ograniczonymi prawami, NoNewPrivileges=true już jest ale to za mało.
S4. Brak lockfile'a (poetry.lock / requirements.lock)
Brak: poetry.lock, requirements.lock, Pipfile.lock, pdm.lock  
Ryzyko: Reprodukowalność buildów = 0. CI może zainstalować inną wersję zależności niż dev.  
Zalecenie: Wybrać jednego managera (poetry lub pip freeze > requirements.lock) i commitować lockfile.
🟠 Poziom 2 — Architektura i techniczny dług
A1. DUPLIKACJA KODU: DWA API, DWIE WARSTWY
Stary (legacy)
interfaces/api.py (747 linii)
agent/agent.py (764 linii)
interfaces/telegram_api.py
Problem: Oba zbiory plików istnieją obok siebie. interfaces/api.py zawiera własną fabrykę FastAPI z własnymi modelami Pydantic, własnym worker loop, własnymi endpointami. api/ zawiera refactored wersję. main.py:23 używa api.app.create_app(), ale stary kod w interfaces/ i agent/ jest wciąż importowany przez nowy kod (core/services.py importuje z agent.tools.*, agent.alerts.*).  
Ryzyko: Utrzymanie dwóch warstw = dwa razy więcej bugów. Stary kod może być bezrefleksyjnie importowany, tworząc niejawny coupling.
A2. Dual persistence (JSON + SQLite)
Plik: agent/state.py:47 USE_SQLITE_STATE  
Agent utrzymuje stan równolegle w SQLite (docelowo) i JSON (fallback). _sync_to_sqlite() (linia 137) i _load_state_from_sqlite() (linia 319), ale load_state() (linia 370) zawsze próbuje SQLITE → jeśli nie ma → fallback do JSON. To znaczy, że wszystkie odczyty stanu przechodzą przez łańcuch: SQLite query → sprawdź None → ewentualnie czytaj JSON. Każdy request = 1-3 zapytań SQL + potencjalnie I/O pliku.
A3. agent/state.py → 1237+ linii
Stanowy moduł zawierający: locki, migrację, operacje JSON, operacje SQLite, walidację invariantów, cleanup, listowanie sesji, backup, DIFF storage, content storage. To woła o podział na 3-4 moduły:
- state/core.py — podstawowe CRUD
- state/locks.py — lock management
- state/migration.py — legacy → SQLite
- state/sync.py — SQLite sync
A4. agent/agent.py (764 linii) vs core/services.py (343 linii)
Stary agent ma wszystko: routing komend, planowanie, generowanie diffów, obsługę błędów. Nowa warstwa core/services.py ma tylko abstrakcje i implementacje serwisów. Logika agenta wciąż w starym pliku. api/app.py:_worker_loop() (linia 118) importuje z agent.agent import process_message — czyli worker loop używa starego agenta, nie nowej warstwy core.
A5. cli/ to pusty szkielet
Plik: cli/__init__.py — tylko docstring. Brak CLI entry point (main() w main.py nie używa cli/).
